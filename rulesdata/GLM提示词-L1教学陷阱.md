### 4. 陷阱一：赋值当等于（.trap-box）

**陷阱标题**：陷阱一：赋值当等于

**严重程度**：🔴 严重（最高级别）

#### 4.1 现象描述

**典型错误代码**（.code-block）：
```cpp
int a = 5;
a = a + 1;  // 学生说：这不对吧？a怎么可能等于a+1？
```

**学生典型反应**：
- "老师，这行代码是错的！"
- "a怎么可能等于a+1？这不科学！"
- "如果a=5，那a=a+1就是5=6，矛盾！"

#### 4.2 原因分析（.alert-tip）

**根本原因**：数学思维 vs 编程思维

> 在数学中，`a = a + 1` 确实是一个矛盾，因为等式两边不可能同时成立。
>
> 但在编程中，`=` 不是"等于"，而是"赋值"。`a = a + 1` 的意思是：
> 1. 先计算右边的 `a + 1`
> 2. 把结果赋给左边的 `a`
> 3. 原来的 `a` 被覆盖了

**认知冲突**：
> 学生学了多年的数学，`=` 就是等于。现在突然告诉ta，`=` 不是等于，是赋值。
> 这种思维转变需要时间和反复练习。

#### 4.3 危害评估（.alert-warn）

**直接影响**：
- 无法理解累加器：`sum = sum + x`
- 无法理解计数器：`count = count + 1`
- 无法写循环（L2的内容）

**长期影响**：
- 到L2学循环时完全跟不上
- 到L3学数组累加时卡住
- 不得不回头补课，浪费大量时间

#### 4.4 解决方案（.step-list）

**第一步：明确区分符号**

> 告诉学生：
> - 数学中的等于用 `=`（两个横线一样长）
> - 编程中的赋值用 `←`（左箭头，表示数据流向左边）
> - C++里用 `=` 表示赋值，只是符号巧合，含义完全不同

**第二步：用生活例子解释**

> "假设你有一个盒子叫a，里面放了5个苹果。
> 现在我说：把a里的苹果数，加上1个，再放回a。
> 这就是 `a = a + 1`。
> 不是问'a等于a+1吗'，而是'把a+1的结果赋给a'。"

**第三步：可视化演示（.sim-table）**

| 步骤 | 代码 | a的值 | 说明 |
|------|------|-------|------|
| 1 | int a = 5; | 5 | 初始化 |
| 2 | a = a + 1; | 6 | 先算右边5+1=6，再赋给左边 |
| 3 | a = a * 2; | 12 | 先算右边6*2=12，再赋给左边 |

**第四步：大量练习（.code-block）**

练习追踪变量值：
```cpp
int x = 3;
x = x + 2;  // x = ?
x = x * 2;  // x = ?
x = x - 5;  // x = ?
```

#### 4.5 预防措施（.checklist-box）

**备课时**：
- [ ] 第一课就明确区分"赋值"和"等于"
- [ ] 准备至少3个生活化的例子
- [ ] 准备至少5道追踪变量值的练习题

**课堂上**：
- [ ] 每节课前5分钟复习赋值的概念
- [ ] 发现学生有疑惑，立即停下澄清
- [ ] 用"盒子"比喻贯穿始终

**作业设计**：
- [ ] 每份作业都包含2-3道赋值追踪题
- [ ] 要求学生写出每一步的变量值

#### 4.6 检验方法

**课堂提问**：
> "如果a=10，执行a=a+5后，a等于多少？"
> - 正确回答：15
> - 错误回答："这不对吧？"

**变体测试（.code-block）**：
```cpp
int a = 5, b = 3;
a = a + b;  // a=?, b=?
b = a - b;  // a=?, b=?
a = a - b;  // a=?, b=?
// 问：最后a和b的值是多少？（答案：a=3, b=5，交换了！）
```

### 5. 陷阱二：忽略类型（.trap-box）

**陷阱标题**：陷阱二：忽略类型

**严重程度**：🟠 较严重

#### 5.1 现象描述

**典型错误代码**（.code-block）：
```cpp
int a = 5, b = 2;
cout << a / b;  // 学生期望输出2.5，实际输出2
```

**学生反应**：
- "老师，程序出bug了！5除以2应该是2.5，为什么输出2？"
- "我的平均成绩计算错了，程序有问题！"

**更隐蔽的版本**（.code-block）：
```cpp
int sum = 85 + 92 + 78;  // sum = 255
double avg = sum / 3;     // 期望85.0，实际得到85（整数除法）
```

#### 5.2 原因分析

**根本原因**：整数除法的截断特性

> C++中，`int / int` 的结果是 `int`，小数部分直接截断（不是四舍五入）。
> 
> 这是C++的设计，不是bug。但很多初学者不知道这一点。

**数学思维的影响**：
> 数学中，5÷2=2.5。学生自然地期望程序也这样。
> 但程序更关心"类型匹配"，而不是"数学正确"。

#### 5.3 危害评估

**直接影响**：
- 计算结果错误
- 平均值、百分比计算出错
- 数学类题目全部失分

**长期影响**：
- 不理解类型转换，L3学数组时混淆
- 不理解内存，L5学指针时崩溃

#### 5.4 解决方案

**第一步：明确解释整数除法**

> "在C++中，整数除以整数，结果还是整数，小数部分直接扔掉。
> 5/2=2，不是2.5。
> 这不是bug，是C++的设计。"

**第二步：展示正确做法（.code-block）**

```cpp
// 方法1：至少一个操作数是double
double result = 5.0 / 2;  // 2.5

// 方法2：强制类型转换
double result = (double)5 / 2;  // 2.5

// 方法3：先转成double再除
double a = 5, b = 2;
double result = a / b;  // 2.5
```

**第三步：总结规则（.alert-tip）**

> 规则：想要小数结果，至少一个操作数要是小数类型（double或float）。

#### 5.5 预防措施

- 专门准备一节课讲整数除法
- 准备10道类型转换练习题
- 让学生养成习惯：计算平均值时，分子先转double

#### 5.6 检验方法（.code-block）

```cpp
// 让学生预测输出
int a = 7, b = 2;
cout << a / b << endl;      // ? (3)
cout << a / 2.0 << endl;    // ? (3.5)
cout << 7.0 / b << endl;    // ? (3.5)
```

### 6. 陷阱三：变量未初始化（.trap-box）

**陷阱标题**：陷阱三：变量未初始化

**严重程度**：🔴 严重（最难发现）

#### 6.1 现象描述

**典型错误代码**（.code-block）：
```cpp
int sum;  // 只声明，未初始化
cin >> a >> b;
sum = a + b;  // 这里是对的
// 但如果在cin之前用了sum...
```

**更隐蔽的错误**：
```cpp
int max;  // 未初始化
cin >> max;  // 输入第一个数
// 然后比较后续输入的数
// 但如果第一个数就是最大值，后面的逻辑可能出错
```

#### 6.2 原因分析

**根本原因**：C++不自动初始化变量

> 在C++中，声明变量时如果不赋值，变量的值是不确定的（随机值，取决于内存中原来有什么）。
>
> 这不是0，不是空，是"垃圾值"。

**为什么难发现**：
> - 有时程序"恰好"能运行（垃圾值碰巧是合理的）
> - 有时程序行为"随机"（每次运行结果不同）
> - 调试时很难定位问题

#### 6.3 危害评估

**直接影响**：
- 程序行为不可预测
- 调试极其困难（"幽灵bug"）
- 同样的代码，今天能跑，明天不能跑

**长期影响**：
- 学生怀疑"程序有问题"、"编译器有问题"
- 养成坏习惯，到L3-L5时bug满天飞

#### 6.4 解决方案

**铁律**：声明变量时必须初始化！

> 不好的习惯：`int a;`
> 好的习惯：`int a = 0;`

**解释原因**：
> "变量就像盒子。声明变量是拿一个盒子，但不初始化就是盒子里可能有垃圾。
> 你永远不知道里面是什么，可能是0，可能是100，可能是-847329。
> 使用前一定要先放一个确定的值进去。"

#### 6.5 预防措施

- 强制执行：不初始化的变量，代码不给过
- 建议使用开启严格警告的编译选项
- 让学生养成肌肉记忆：写 `int a` 时，手指自动敲 ` = 0`

#### 6.6 检验方法（.code-block）

```cpp
// 让学生预测这个程序的输出（实际上无法预测）
int x;
cout << x << endl;  // 可能是0，可能是任意值
```

### 7. 陷阱四：混淆输入输出（.trap-box）

**陷阱标题**：陷阱四：混淆输入输出

**严重程度**：🟡 中等

#### 7.1 现象描述

**典型错误代码**（.code-block）：
```cpp
int a;
cout << "请输入a的值：";  // 提示信息
cout << a;               // 错误！应该是cin >> a
```

**或者**：
```cpp
int a, b;
cin >> a >> b;           // 程序在这里等待输入
// 但学生不知道，以为程序卡住了
```

#### 7.2 原因分析

**根本原因**：cin/cout概念混淆

> - `cin` 是输入（从键盘读数据到程序）
> - `cout` 是输出（从程序输出数据到屏幕）
>
> 学生容易搞混箭头方向：
> - `cin >> a`：数据流向a（从键盘流入变量）
> - `cout << a`：数据从a流出（从变量流出到屏幕）

#### 7.3 解决方案

**记忆法**：
> "数据流向变量用 `>>`（像漏斗流入）
> 数据从变量流出用 `<<`（像喷泉流出）
> 
> 或者记住：
> - `cin` 后面用 `>>`（流入）
> - `cout` 后面用 `<<`（流出）"

**养成提示习惯**：
```cpp
int a;
cout << "请输入a的值：";  // 一定要先提示
cin >> a;                  // 然后再输入
```

#### 7.4 预防措施

**模板化**：给学生固定模板
```cpp
// 输入模板
cout << "请输入xxx：";
cin >> xxx;
```

#### 7.5 检验方法

检查每个输入是否有对应的提示，箭头方向是否正确。
